# book2ppt

## 这个 Skill 做什么

把一本书的 PDF 转化为线下 book club 分享演讲幻灯片。

不是书的内容摘要，而是设计一场对话：提炼书的核心框架，加入分享者的独立判断，以引发现场讨论收尾。输出为 HTML 幻灯片，使用 `beautiful-html-templates` 视觉系统。

## 何时使用

- 用户输入书籍 PDF，想做成线下分享演讲
- 场景是 book club、公司读书会、课程导读、活动分享
- 用户说"帮我把这本书做成 PPT"、"书籍分享"、"读书会演讲"

## 前置条件

- `beautiful-html-templates` 已安装：`~/.claude/skills/beautiful-html-templates/`

## 工作流

### Step 1 · 读书（根据页数分策略）

**判断页数**：先读 PDF 第一页或目录，估算总页数。

#### ≤ 300 页：全文读取

分段读取全书，每次使用 `pages` 参数批量读取。重点关注：
- 序言 / 前言 / 引言
- 每章开头与结尾
- 结语 / 后记
- 标有定义、框架、模型、案例、数据的段落

#### ≥ 400 页：战略性采样

按优先级分层读取：

| 优先级 | 读什么 | 目的 |
|---|---|---|
| 必读 | 目录（TOC） | 理解全书结构 |
| 必读 | 序言 / 前言（前 5%） | 作者意图 + 核心论点 |
| 必读 | 结语 / 后记（后 5%） | 作者总结 + 行动建议 |
| 重要 | 每章开头 2 页 | 该章核心主张 |
| 重要 | 每章最后 1 页 | 该章小结 |
| 按需 | 用户指定章节 | 深度展开 |

#### PDF 提取降级方案

Read 工具有时无法解析 PDF（加密、格式异常、环境限制）。按以下顺序降级：

1. **Read 工具**（带 `pages` 参数）→ 优先尝试
2. **pdfplumber**（系统 Python）→ Read 失败时执行：
   ```bash
   /opt/homebrew/bin/python3 - <<'EOF'
   import pdfplumber
   with pdfplumber.open("/path/to/book.pdf") as pdf:
       for i, page in enumerate(pdf.pages[:10], 1):
           print(f"--- Page {i} ---")
           print(page.extract_text() or "")
   EOF
   ```
   若 pdfplumber 未安装：`/opt/homebrew/bin/python3 -m pip install pdfplumber --break-system-packages`
3. **用户手动粘贴** → 两种方法均失败时，告知用户粘贴关键章节文本或目录，继续后续步骤

读完后进入 Step 2。

---

### Step 2 · 书籍解析（7 个维度）

从读取内容中提取以下 7 个维度，为内容简报做准备：

```
① 核心论点      全书最核心的一句话主张（不超过 30 字）
② 作者背景      为什么这个人有资格写这本书（2-3 句）
③ 核心框架      书中提出的 2-4 个可复用心智模型或方法论
④ 支撑证据      每个框架的关键数据、实验、案例（各 1-2 个）
⑤ 金句摘录      最值得单独呈现的 3-5 句话（尽量保留原文）
⑥ 批判视角      这本书的亮点 / 局限 / 作者漏掉了什么
⑦ 行动建议      选 1-2 个角度深入讲解，每个具体到「今晚可以做的第一步」；不列 3-5 条抽象清单
```

提取完成后进入 Step 3。

**内容自检（展示前主动修正）**：

| # | 检验项 |

### Step 5.5 · 逐页结构确认（用户交互）

模板选定后，**先输出逐页清单，等用户确认，再写 HTML**。

格式：每行一张幻灯片，包含页码、页面标题、布局意图（一句话说清排什么）。**同时在括号内注明排版预算**，防止后续生成时靠感觉选字号：

```
01 Cover         — hero: 书名大字 + 作者标注 + 标语右下角
02 The Problem   — red: 大字 2 行 + 小字 1 行
                  （大字 72px → max-width:none → 每行约 28 字符上限）
03 About Author  — 两栏: 照片左 / 简介+数据右
                  （照片固定 210×280px；右栏预估 ~300px 高）
04 1% Rule       — stat: 37× 大字 + 两行算式 + 说明小字
05 The Paradox   — red: 大字 3 行（每行 ≤ 22 字符 @ 64px）
06 Why Goals Fail— 两栏: 左=原则短句（中）/ 右=工具卡片（中）
                  （左右预估行数相差 ≤ 2 行）
...
xx Close         — 结束页: 大字口号 + 推荐语 + 链接
```

**排版预算三项规则**（每张幻灯片标注时对照）：

| 幻灯片类型 | 需要标注的内容 |
|---|---|
| 含大字引言/口号 | 计划字号上限 + 每行最多字符数（= available_width ÷ (font_size × 0.65)） |
| 两栏布局 | 左栏内容量（短/中/长）/ 右栏内容量，两侧不应相差 > 2 行 |
| 含图片/照片 | 图片指定 px 尺寸（基于相邻内容块预估高度） |

**这一步的目的**：在写任何 HTML 之前，对齐每页的核心信息点、布局意图和排版边界。用户如需调整结构，在此处修改，进入 Step 6 后不再做结构性重构。

用户确认后进入 Step 6。

### Step 6 · 生成完整幻灯片

#### 6.0 每张幻灯片开写前的三项预检

**在写任何一张幻灯片的 HTML 之前**，先做以下三项判断。不做完不动手。

**① 字符上限验证（大字幻灯片必做）**

```
每行最多字符数 = available_width ÷ (font_size × 0.65)
```

- `available_width`：若有 `max-width` 限制，取 `max-width`；否则取 `viewport_width − padding`（通常 padding = 112px）
- `0.65` 是 Shrikhand 等展示字体的平均字宽系数（字母宽 ≈ 字号 × 0.65）
- 用 `<br>` 控制换行时，每段文字**单独数字符数**并与上限对比

超限时按优先级处理：**缩短文字** → **降低字号** → **去掉 max-width 改用全宽**。禁止"先写下来再等用户报孤行"。

示例：`max-width:900px, font-size:90px` → 每行最多 **15 字符**。"Each action you take"（20 字符）必然折行，"take" 会孤行。

**② 两栏高度预估（两栏布局必做）**

写之前估算左右栏各约几行内容（按主字号折算）：

- 差值 ≤ 2 行 → 可以直接写
- 差值 > 2 行 → 先裁短较长侧，或拆分为两张幻灯片

不依赖事后用户反馈来平衡，写之前就处理好。

**③ 视觉层级判断（每张幻灯片必做）**

开写前先问：*哪句话是主旨，哪句话是例证？*

- **主旨** → 最大字号，占视觉焦点
- **例证**（比喻、故事、案例）→ 最小字号，辅助说明

不允许例证的字号 ≥ 主旨字号。层级颠倒会让听众记住比喻而忘掉观点。

#### 6.2 幻灯片弧线（默认参考，以 Step 5.5 用户确认的清单为准）

以下为默认弧线，实际结构以 Step 5.5 中用户确认的逐页清单为准，此处仅供初始规划参考，不要照搬。

```
Section A · 引入（3 页）
  01 Cover          书名 / 作者 / 分享者姓名
  02 Why This Book  这本书解决什么问题 / 为什么现在值得读
  03 About Author   作者背景 + 这本书的写作缘起

Section B · 核心思想（每个框架 2 页）
  04 Big Idea       全书核心论点（1 页，视觉冲击力最强）
  05 Framework 1    框架名称 + 概述
  06 Framework 1    案例 / 数据 / 实际应用
  07 Framework 2    框架名称 + 概述
  08 Framework 2    案例 / 数据 / 实际应用
  （按框架数量循环，2-4 个框架）

Section C · 分享者视角（2-3 页）
  xx Key Quotes     2-3 句最值得记住的原书话（每句独占或共享 1 页）
  xx Critique       亮点 + 局限（可选——公司内训等场合可删）
  xx My Takeaway    我打算怎么用它（第一人称，具体）

Section D · 引发讨论（可选，小圈子 book club 适用，大型演讲或内训可删）
  xx Discussion     3-4 个讨论问题（有争议性，能引发观点碰撞）
  xx Close          推荐指数 / 适合谁读
```

#### 6.3 内容填充原则

- **保留**：字体 / 配色 / CSS 类名 / 装饰元素 / 导航逻辑
- **替换**：标题 / 正文 / 数据 / 作者日期
- 每页只传递一个核心信息点
- 金句页：原文 + 出处，不改写，不缩短
- Critique 页：分享者自己的判断，要有立场，不做"各有利弊"的外交辞令
- Discussion 页：问题要有争议性，能让人想回答又没有标准答案

#### 6.4 作者页：必须尝试附作者照片

About Author 页应尽量放作者本人的真实照片，而非空白头像或纯文字。

**查找流程**：

1. 先用 `curl -I` 检查 Wikipedia Commons 是否有该作者图片：
   ```
   https://upload.wikimedia.org/wikipedia/commons/<路径>/<文件名>.jpg
   ```
2. 搜索策略（按顺序尝试）：
   - Wikipedia 作者页面 → Infobox 中的图片链接
   - Wikimedia Commons 搜索作者姓名
   - 作者官网 / 出版社官方页面（仅限 `<img>` 直链可用时）
3. 找到有效 URL 后，写入 `<img src="..." alt="作者名">` 并**必须加 `onerror` fallback**：
   ```html
   <img src="<photo-url>" alt="<作者名>"
        onerror="this.style.display='none';this.nextElementSibling.style.display='flex'">
   <div class="author-avatar-fallback" style="display:none">
     <span class="author-initials"><作者名首字母></span>
   </div>
   ```
4. 若多次尝试均无法找到可用图片，fallback 显示作者姓名首字母缩写（如 `JC`），并在交付时告知用户。

**注意**：只使用可直接 `<img src>` 引用的公开链接（Wikipedia Commons 永久链接最可靠）。不下载图片到本地，不引用需要登录或 JS 渲染才能访问的页面。

### Step 6.5 · 视觉自检（生成后逐页检查）

用 `open` 打开，逐页检查以下项目，发现问题立刻修复：

**版面结构**

| # | 检验项 | 判断标准 |
|---|---|---|
| B1 | 版面覆盖率 | 内容铺满页面纵向 60% 以上，无明显空白半区 |
| B2 | 垂直居中 | 内容整体视觉居中，不靠顶不下沉 |
| B3 | 文字可见性 | 所有文字在当前背景色上清晰可读 |
| B4 | 金句页 | 引号内是原书文字，字体够大，来源标注清楚 |
| B5 | 讨论页 | 问题编号清晰，间距足够，台上容易念出来 |
| B6 | 作者页照片 | 有真实作者照片且正常显示；若图片加载失败，fallback 显示姓名首字母而非破图 |

**排版细节**

| # | 检验项 | 判断标准 |
|---|---|---|
| T1 | 孤行 | 无单个单词独占一行；句尾最后一行至少有 2 个词 |
| T2 | 字号层级 | 每页可辨别三级：核心大字 / 副标题中字 / 说明小字，级差肉眼可感知 |
| T3 | 演讲者小字 | 正文区只放关键词触发词，不出现完整段落 |
| T4 | 语言一致 | 全页英文或全页中文，不混排 |
| T5 | 数学符号 | 乘号用 `×`（U+00D7），约等于用 `≈`，不等号用 `≠`；指数用 `<sup>` 而非 Unicode 上标字符 |
| T6 | 折行验证 | 每个 `<br>` 段落的字符数 ≤ `available_width ÷ (font_size × 0.65)`；若无 max-width 则 available_width = viewport − padding（≈ 112px）。超限说明该行在该字号下会再次折行，必须缩短文字或降字号 |
| T7 | 两栏平衡 | 目测左右栏内容行数之差 ≤ 2 行；含图片时，图片高度与相邻内容块高度视觉匹配（不应相差超过 30%） |

**逻辑表达**

| # | 检验项 | 判断标准 |
|---|---|---|
| L1 | 符号位置 | `=` / `≠` / `→` 紧邻它所连接的两个元素，不能悬空 |
| L2 | 行动建议深度 | 选 1–2 个角度深入讲解，而非 5 个抽象 bullet；每条建议可以当天执行 |
| L3 | 场合标签 | 幻灯片内不出现"读书沙龙"/"公司内训"等具体场合标注，不出现日期 |

## 设计通则（从实战中提炼，出现过的问题不能重犯）

### 1 · 孤行零容忍
正文中任何一行不能只有一个单词或两三个字符。发现后用 `<br>` 在上一个自然断点手动换行，或收窄 `max-width` 让自然折行落在更好的位置。

### 2 · 三级字号，级差明显
每张幻灯片必须有可识别的三级字号：
- **核心大字**（标题/数据）—— 视觉焦点，占画面
- **副标题/关键词**（中字）—— 补充说明大字
- **说明小字**（正文/子弹点）—— 背景信息

三级之间差距至少 1.5×，否则层级感消失。

### 3 · 留白要用 CSS 消除，不能靠猜
竖向 flex 布局若子元素没撑满，根本原因通常是：
- 父容器缺少 `align-items: stretch`（子元素不拉伸）
- 子容器缺少 `flex: 1`（不填充剩余空间）

不要调 padding 去"补"空白，要找根本原因改 flex 属性。

### 4 · 符号必须紧贴它所连接的内容
`=`、`≠`、`→` 等逻辑符号的位置要在视觉上紧邻它连接的两个元素。如果两者不在同一行，符号就失去意义。拿不准就拆成两行分别呈现对比关系。

### 5 · 演讲者小字 ≠ 演讲稿
幻灯片上的小字只起"提词"作用——几个关键词让演讲者看到就能展开。完整的句子、段落属于演讲稿，不属于幻灯片。

### 6 · 语言在开工前确定，不在中途改
全英或全中，开工前一次决定。中途改语言意味着所有幻灯片都要重检字体、间距和对齐，代价很高。

### 7 · 数学与特殊符号使用规范
| 场景 | 正确写法 | 错误写法 |
|---|---|---|
| 乘号 | `×`（U+00D7）| 字母 `x` |
| 指数 | `<sup>365</sup>` | Unicode 上标 `³⁶⁵` |
| 约等于 | `≈` | `~` |
| 不等号 | `≠` | `!=` |

display 字体（如 Shrikhand）对 Unicode 上标字符渲染不可控，一律用 HTML 标签。

### 8 · 绝对定位元素第一轮就确认位置
角落标注、装饰标语等绝对定位元素，生成后立刻在浏览器截图验证 `top/bottom/left/right` 数值，不要靠文字描述反复猜。一次确认，后续不动。

### 9 · 大字折行：先算后写，不靠感觉选字号

字号确定前必须先计算：

```
每行字符数 = available_width ÷ (font_size × 0.65)
```

`available_width`：有 `max-width` 时取 `max-width`；否则取 `viewport_width − padding`（通常 padding ≈ 112px）。

若计划文字超出每行字符数，按优先级处理：**缩短文字** → **降字号** → **去掉 max-width 改用全宽**。

禁止"先写下来，再等用户报告孤行"。孤行是排版计算失误，不是用户验收任务。

### 10 · 图片尺寸永远写 px，不用比例自动推算

`aspect-ratio + width:100%` 在流体 Grid 列里尺寸不可预测（列宽变则图片高度剧变）。凡是图片/照片：

1. 先估算相邻内容块的预期高度
2. 直接写 `width:Xpx; height:Ypx`，覆盖 class 里的 `width:100%`

### 11 · 两栏幻灯片：写之前数行数，不靠事后修

写两栏 HTML 前先估算：左栏约几行 × 字号，右栏约几行 × 字号。差值 > 2 行时主动平衡（裁短较长侧或拆页），不等用户反馈"左边太长了"再改。

### 12 · 主旨与例证的视觉层级不能颠倒

每张幻灯片开写前先判断层级：

- **主旨**（核心观点、数据结论）→ 最大字号
- **例证**（比喻、故事、案例）→ 最小字号，配注位置

比喻是帮助理解的工具，不是主角。若比喻占了最大字号，听众会记住比喻而忘掉观点。

### 13 · 子类 CSS 必须显式覆盖父类的所有布局属性

写派生 slide 类（如 `.slide-pillars`、`.slide-hero`）时，必须对照基类 `.slide` 检查所有布局属性：

```
justify-content / align-items / align-content
flex-direction / padding / overflow
```

父类属性默认继承，不写不代表不生效。遗漏一个就可能导致内容意外居中、飘移或被裁剪。

### 14 · 每张幻灯片只讲一个中心思想

**这是所有规则里优先级最高的一条。**

一张幻灯片只能传递一个独立概念。出现以下任意情况即为违规：

- 同一页列了多个技巧、方法、工具（"方法一 + 方法二 + 方法三"）
- 同一页出现了两个等重的内容块（卡片、列）
- 页面标题可以用"和"连接两个名词

处理方式：合并成一个主旨，或拆为两张幻灯片。**不允许"精简合并"来绕开拆页**——把三个卡片压缩成三行子弹点仍然是三个概念，不是一个。

## 与 beautiful-doc2ppt 的核心差异

| 项目 | beautiful-doc2ppt | book2ppt |
|---|---|---|
| 输入 | 任意文档 | 书籍 PDF |
| PDF 读取 | 全文 | ≤300 页全读，≥400 页采样 |
| 内容提取 | 原文结构映射 | 7 维度重构 |
| 分享者声音 | 无 | 必须有（Critique + My Takeaway） |
| 金句 | 装饰性 | 独立幻灯片，保留原文 |
| 结尾 | 行动项 | 讨论问题 + 推荐指数 |
| 叙事弧线 | 业务汇报结构 | 书评 / 说服结构 |
